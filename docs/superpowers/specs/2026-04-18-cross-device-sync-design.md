# Cross-Device Sync — Design Spec
_2026-04-18_

## Overview

Replace localStorage with Supabase (Postgres) so fueling plans are accessible from any device. The browser calls Supabase's PostgREST API directly using `fetch()` — no backend server, no SDK dependency. The app continues to be deployed as a static site on Netlify.

**Constraints:**
- No user accounts (single user for now; schema is multi-user ready)
- Connectivity required — no offline mode
- Zero new JS dependencies (direct fetch, no Supabase SDK)
- `index.html` and `style.css` are unchanged except minimal loading/toast markup and styles

---

## Architecture

```
Browser (Netlify static site)
  └── data.js  ──fetch()──▶  Supabase PostgREST API  ──▶  Postgres
```

Three constants in `data.js`:
- `SUPABASE_URL` — Supabase project URL (`https://xxxx.supabase.co`)
- `SUPABASE_ANON_KEY` — public anon key (designed to be in client code)
- `USER_ID` — a UUID generated once and hardcoded; scopes all data to this user

A single `supabaseRequest(method, path, body)` helper handles auth headers and JSON encoding for all fetch calls. All `data.js` public functions become async.

`recentProducts` is intentionally left in localStorage — it is a per-device UI convenience (recently used on phone ≠ recently used on desktop), not core plan data.

## Database Schema

All tables defined in Supabase SQL editor. IDs are UUIDs generated by Postgres.

```sql
-- Single row for now. Multi-user: real auth users go here.
CREATE TABLE users (
  id UUID PRIMARY KEY
);

CREATE TABLE products (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           UUID NOT NULL REFERENCES users(id),
  brand             TEXT,
  name              TEXT NOT NULL,
  type              TEXT NOT NULL,
  carbs_per_unit    INTEGER NOT NULL DEFAULT 0,
  sodium_per_unit   INTEGER NOT NULL DEFAULT 0,
  caffeine_per_unit INTEGER NOT NULL DEFAULT 0,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE events (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID NOT NULL REFERENCES users(id),
  name       TEXT NOT NULL,
  date       DATE,
  type       TEXT NOT NULL DEFAULT 'other',
  notes      TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE segments (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id          UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  name              TEXT NOT NULL,
  duration_hours    REAL NOT NULL DEFAULT 1,
  carbs_per_hour    INTEGER NOT NULL DEFAULT 0,
  sodium_per_hour   INTEGER NOT NULL DEFAULT 0,
  caffeine_per_hour INTEGER NOT NULL DEFAULT 0,
  sort_order        INTEGER NOT NULL DEFAULT 0,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE items (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  segment_id        UUID NOT NULL REFERENCES segments(id) ON DELETE CASCADE,
  product_id        UUID REFERENCES products(id) ON DELETE SET NULL,
  name              TEXT NOT NULL,
  brand             TEXT,
  type              TEXT NOT NULL,
  carbs_per_unit    INTEGER NOT NULL DEFAULT 0,
  sodium_per_unit   INTEGER NOT NULL DEFAULT 0,
  caffeine_per_unit INTEGER NOT NULL DEFAULT 0,
  quantity          INTEGER NOT NULL DEFAULT 1,
  sort_order        INTEGER NOT NULL DEFAULT 0,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**Key invariants preserved:**
- `items.product_id` is nullable — one-off items (not saved to library) have no product reference
- `items` snapshots all product values at add-time; editing/deleting a library product never affects existing plans
- `segments.sort_order` and `items.sort_order` preserve ordering (Postgres does not guarantee insertion order)
- Cascade deletes: removing an event removes all its segments and items automatically

## `save_event` RPC Function

Saving an event requires updating the event row, deleting existing segments (cascading to items), and re-inserting segments and items. These must be atomic — a partial failure would leave the database inconsistent.

Solution: a Postgres function called via Supabase RPC (`POST /rest/v1/rpc/save_event`). Postgres functions run inside a transaction; if any step fails, the whole operation rolls back.

```sql
CREATE OR REPLACE FUNCTION save_event(event_data jsonb)
RETURNS void AS $$
DECLARE
  evt_id UUID;
  seg jsonb;
  seg_id UUID;
  itm jsonb;
BEGIN
  -- Upsert event row
  INSERT INTO events (id, user_id, name, date, type, notes)
  VALUES (
    (event_data->>'id')::UUID,
    (event_data->>'user_id')::UUID,
    event_data->>'name',
    NULLIF(event_data->>'date', '')::DATE,
    COALESCE(event_data->>'type', 'other'),
    event_data->>'notes'
  )
  ON CONFLICT (id) DO UPDATE SET
    name       = EXCLUDED.name,
    date       = EXCLUDED.date,
    type       = EXCLUDED.type,
    notes      = EXCLUDED.notes,
    updated_at = now();

  evt_id := (event_data->>'id')::UUID;

  -- Remove existing segments (cascades to items)
  DELETE FROM segments WHERE event_id = evt_id;

  -- Re-insert segments and items
  FOR seg IN SELECT * FROM jsonb_array_elements(event_data->'segments')
  LOOP
    seg_id := (seg->>'id')::UUID;

    INSERT INTO segments (id, event_id, name, duration_hours,
                          carbs_per_hour, sodium_per_hour, caffeine_per_hour, sort_order)
    VALUES (
      seg_id,
      evt_id,
      seg->>'name',
      (seg->>'durationHours')::REAL,
      COALESCE((seg->'targets'->>'carbsPerHour')::INTEGER, 0),
      COALESCE((seg->'targets'->>'sodiumPerHour')::INTEGER, 0),
      COALESCE((seg->'targets'->>'caffeinePerHour')::INTEGER, 0),
      COALESCE((seg->>'sortOrder')::INTEGER, 0)
    );

    FOR itm IN SELECT * FROM jsonb_array_elements(seg->'items')
    LOOP
      INSERT INTO items (id, segment_id, product_id, name, brand, type,
                         carbs_per_unit, sodium_per_unit, caffeine_per_unit,
                         quantity, sort_order)
      VALUES (
        (itm->>'id')::UUID,
        seg_id,
        NULLIF(itm->>'productId', 'null')::UUID,
        itm->>'name',
        itm->>'brand',
        itm->>'type',
        COALESCE((itm->>'carbsPerUnit')::INTEGER, 0),
        COALESCE((itm->>'sodiumPerUnit')::INTEGER, 0),
        COALESCE((itm->>'caffeinePerUnit')::INTEGER, 0),
        COALESCE((itm->>'quantity')::INTEGER, 1),
        COALESCE((itm->>'sortOrder')::INTEGER, 0)
      );
    END LOOP;
  END LOOP;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Grant execute to the anon role so PostgREST can call it
GRANT EXECUTE ON FUNCTION save_event(jsonb) TO anon;
```

The function uses `SECURITY DEFINER` (runs as DB owner, bypassing RLS) so it can delete and re-insert across tables in one transaction. Because RLS is bypassed, the function must validate the caller explicitly — add this check at the top of the function body:

```sql
-- Reject any attempt to write to a different user's data
IF (event_data->>'user_id')::UUID != 'YOUR-USER-UUID'::UUID THEN
  RAISE EXCEPTION 'unauthorized';
END IF;
```

## API Layer (`data.js`)

### Helper

```js
async function supabaseRequest(method, path, body) {
  var res = await fetch(SUPABASE_URL + '/rest/v1/' + path, {
    method: method,
    headers: {
      'apikey': SUPABASE_ANON_KEY,
      'Authorization': 'Bearer ' + SUPABASE_ANON_KEY,
      'Content-Type': 'application/json',
      'Prefer': method === 'POST' ? 'return=representation' : ''
    },
    body: body ? JSON.stringify(body) : undefined
  });
  if (!res.ok) throw new Error(await res.text());
  return res.status === 204 ? null : res.json();
}
```

### Key/normalisation convention

Postgres columns use `snake_case`; the JS app uses `camelCase`. Two small functions convert on read and write:

- `dbToProduct(row)` / `productToDb(p)` — flat object, trivial mapping
- `dbToEvent(row)` / `eventToDb(e)` — nested; segments and items also normalised

Supabase nested select (`select=*,segments(*,items(*))`) returns a fully nested object in one request, so `getEvents()` and `getEvent(id)` are single fetch calls.

### Public interface (all functions become async)

```js
// Products
async getProducts()           // GET /products?user_id=eq.USER_ID&select=*
async saveProduct(p)          // POST /products with upsert header
async deleteProduct(id)       // DELETE /products?id=eq.ID

// Events
async getEvents()             // GET /events?user_id=eq.USER_ID&select=*,segments(*,items(*))
async getEvent(id)            // GET /events?id=eq.ID&select=*,segments(*,items(*))
async saveEvent(evt)          // POST /rpc/save_event
async deleteEvent(id)         // DELETE /events?id=eq.ID (cascades)

// Recent products — unchanged, still localStorage
getRecentProducts()
recordProductUsed(id)
```

## `app.js` changes

All render and event-handler functions that call data functions become `async` and `await` the result. The pattern is consistent:

```js
// Before
function renderEvents() {
  var events = Data.getEvents();
  ...
}

// After
async function renderEvents() {
  var events = await Data.getEvents();
  ...
}
```

`navigate()` calls `renders[view]()` and attaches `.catch()` to surface errors without crashing the app.

### Loading states

Each view that fetches data on entry shows a brief loading indicator while the request is in flight, replacing it with content on resolve. Implemented as a shared `showLoading(containerId)` / `hideLoading(containerId)` utility in `app.js`.

### Error states

If any data call throws (network failure, Supabase outage), a lightweight toast notification appears at the bottom of the screen: "Couldn't save — check your connection." Toasts auto-dismiss after 4 seconds. Implemented as a shared `showToast(message)` utility.

## localStorage Migration

On first load, `data.js` runs `migrateIfNeeded()`:

1. Check `localStorage.getItem('fuelPlanner.migrated')` — if `'true'`, skip entirely
2. Query Supabase: if the user already has events or products, skip (data came from another device)
3. Read events and products from localStorage
4. If both are empty, write `fuelPlanner.migrated = true` and return — nothing to migrate
5. Insert all products via `saveProduct()`
6. Insert all events via `saveEvent()` RPC
7. On full success: set `fuelPlanner.migrated = true` in localStorage
8. On any failure: leave localStorage untouched, surface a toast — user's data is safe, migration will retry on next load

Migration is silent if successful. It runs before the first view renders so the user never sees a partially migrated state.

## Security

**RLS policies (applied to all tables):**

```sql
-- Enable RLS
ALTER TABLE products ENABLE ROW LEVEL SECURITY;
ALTER TABLE events   ENABLE ROW LEVEL SECURITY;
ALTER TABLE segments ENABLE ROW LEVEL SECURITY;
ALTER TABLE items    ENABLE ROW LEVEL SECURITY;

-- Anon key can only touch rows belonging to USER_ID
CREATE POLICY "personal_access" ON products
  USING (user_id = 'YOUR-USER-UUID'::UUID);

CREATE POLICY "personal_access" ON events
  USING (user_id = 'YOUR-USER-UUID'::UUID);

-- Segments and items are accessed via cascade/join — restrict by parent ownership
CREATE POLICY "personal_access" ON segments
  USING (event_id IN (SELECT id FROM events WHERE user_id = 'YOUR-USER-UUID'::UUID));

CREATE POLICY "personal_access" ON items
  USING (segment_id IN (SELECT id FROM segments
    WHERE event_id IN (SELECT id FROM events WHERE user_id = 'YOUR-USER-UUID'::UUID)));
```

The anon key and USER_ID are in client-side JS source. Anyone who finds them can query Supabase but RLS restricts them to your rows — which contain only fueling plans, not sensitive data. Acceptable for a personal app.

**Multi-user migration path (future):**
- Add Supabase Auth
- Replace hardcoded `USER_ID` with `auth.uid()` in RLS policies
- No schema changes required — `user_id` column already exists on every table

## Out of Scope

- Offline support / sync queue
- Conflict resolution (last write wins implicitly; single active session assumed)
- User accounts / authentication
- Real-time sync between two open tabs/devices simultaneously
